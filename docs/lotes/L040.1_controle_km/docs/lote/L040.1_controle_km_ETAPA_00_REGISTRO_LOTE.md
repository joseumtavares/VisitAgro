## 4.2 Contexto

- **Problema observado:** o módulo **Controle de KM** está registrado como pendente no roadmap/status do VisitAgro. Existe referência de schema para a tabela `km_logs`, mas não há rota API nem página de dashboard dedicadas para o controle diário de quilometragem dos representantes.
- **Causa suspeita:** a base de dados foi parcialmente preparada (`km_logs` existe no schema de referência), porém o contrato API ↔ UI ainda não foi formalizado/implementado. Além disso, o schema atual de `km_logs` precisa ser verificado pelo próximo agente porque não apresenta coluna `workspace`, embora o padrão operacional do projeto exija filtros explícitos de tenancy quando `getAdmin()` usa service role.
- **Impacto:** representantes não conseguem registrar KM diário de visitas; gestores não conseguem auditar deslocamentos por representante/período; relatórios futuros de produtividade, custo por KM e reembolso ficam sem base confiável.
- **Base documental usada:**
  - `AGENTES.md`: confirma Next.js 14 App Router, rotas em `src/app/api/**/route.ts`, Supabase via `getAdmin()`, tenancy por `x-workspace`, necessidade de `deleted_at` em leituras e diff mínimo em arquivos sensíveis.
  - `docs/playbook-operacional.md`: exige lote pequeno, evidência real, DoR/DoD, validação mínima, fluxo ChatGPT → Cursor → Claude.
  - `README.md`: registra o sistema como Next.js 14 + Supabase e marca **Controle de KM — Pendente**.
  - `docs/index.md`: centraliza documentação, patches SQL, lotes e changelog.
  - `docs/changelog.md`: mostra L037/L038 e próximos itens planejados, incluindo controle de KM.
  - `docs/ui/responsividade.md`: exige mobile-first, validação em 375px/768px/≥1024px, cards ou scroll controlado em tabelas, formulário em coluna única e botões com 44px.
  - `docs/patches/L038_responsividade_crud.md`: referência prática para CRUD mobile com cards, tabela desktop preservada e validação funcional.
  - `sql/schema_atual_supabase.sql`: contém `CREATE TABLE public.km_logs` com campos `user_id`, `data`, `veiculo`, `km_ini`, `km_fim`, `percorrido`, `combustivel`, `consumo`, `litros`, `custo_por_km`, `obs`, `deleted_at`, mas sem `workspace`.
  - `src/app/api/`: lista rotas existentes e não contém rota `km-logs`.
  - `src/app/dashboard/`: lista páginas existentes e não contém página `km`.
  - `src/types/index.ts`: contém tipos compartilhados de usuários, representantes, visitas, vendas e comissões; ainda não contém tipo específico `KmLog`.
  - `docs/padrao_de_comentarios.md`: arquivo real encontrado está em v3.0; a solicitação cita v2.0. A implementação futura deve seguir o arquivo real do repositório e registrar esse ajuste documental, sem bloquear o lote.

## 4.3 Escopo

**Incluído:**

- Confirmar e, se necessário, propor migration incremental para `km_logs`, sem reescrever o schema inteiro.
- Garantir contrato mínimo da entidade `KmLog`:
  - `id`
  - `workspace` se confirmado necessário para tenancy direta
  - `user_id`
  - `data`
  - `veiculo`
  - `km_ini`
  - `km_fim`
  - `percorrido`
  - `combustivel`
  - `litros`
  - `consumo`
  - `custo_por_km`
  - `obs`
  - `created_at`
  - `updated_at`
  - `deleted_at`
- Definir regra de cálculo: `percorrido = km_fim - km_ini`.
- Definir validação obrigatória: `km_fim >= km_ini`.
- Definir regra diária: 1 registro ativo por `workspace + user_id + data`, salvo decisão técnica documentada em contrário.
- Criar contrato futuro para API:
  - `GET /api/km-logs` com filtros por período, representante e contexto de usuário.
  - `POST /api/km-logs` para criar registro diário.
  - `PUT /api/km-logs/[id]` para editar registro existente.
  - `DELETE /api/km-logs/[id]` como soft delete via `deleted_at`.
- Definir regra de acesso:
  - `representative`: vê e altera apenas seus próprios registros.
  - `admin` e `manager`: veem registros do workspace e podem filtrar por representante.
  - `user`: sem acesso administrativo; comportamento deve ser confirmado pelo mapa de impacto, preferencialmente acesso apenas próprio ou bloqueio documentado.
- Criar UI futura em `src/app/dashboard/km/page.tsx`:
  - formulário de lançamento diário;
  - listagem por período;
  - filtro de data inicial/final;
  - filtro de representante para admin/manager;
  - cards mobile e tabela desktop;
  - ações editar/remover;
  - totais do período: KM percorrido, litros, combustível/custo quando preenchidos.
- Adicionar item de navegação no dashboard para o módulo de KM, preservando regras atuais de perfil.
- Atualizar documentação:
  - `docs/patches/060_controle_km_logs.md`
  - `docs/index.md`
  - `docs/changelog.md`
  - registro do lote em `docs/lotes/` ou equivalente do fluxo real do repositório.
- Seguir `docs/padrao_de_comentarios.md` real do repositório, com tags `CRITICAL`, `AI-CONTEXT`, `AI-CONTRACT` e `AI-RULE` quando houver risco de tenancy, service role, soft delete ou contrato API/UI.

**Excluído explicitamente:**

- Rastreamento GPS em tempo real.
- Cálculo automático de rota percorrida por mapa.
- Integração automática com check-in/check-out de visitas.
- Reembolso financeiro, aprovação de despesas ou fechamento de folha.
- PDF, WhatsApp, exportação Excel/CSV.
- Relatórios analíticos avançados fora dos totais simples da tela.
- Alteração do módulo de visitas além de leitura documental.
- Refatoração de auth, middleware, Zustand, `apiFetch` ou Supabase admin.
- Mudanças em comissões de representantes.
- Alterações em pedidos, clientes, produtos, indicadores ou pré-cadastros.
- RLS no Supabase; o padrão atual usa service role e filtros explícitos na aplicação.

## 4.7 Validação mínima

1. **Criação de KM diário — representative**
   - Contexto de entrada: usuário autenticado com role `representative`, sem registro ativo para a data atual.
   - Ação: enviar `POST /api/km-logs` com `data`, `veiculo`, `km_ini=1000`, `km_fim=1048`, `litros=6`, `combustivel=36`, `obs`.
   - Resultado esperado: HTTP 201; resposta contém `km_log.percorrido = 48`; `user_id` igual ao usuário autenticado; registro aparece na listagem do próprio representante.

2. **Validação de KM final menor que inicial**
   - Contexto de entrada: usuário autenticado com role `representative`.
   - Ação: enviar `POST /api/km-logs` com `km_ini=1048` e `km_fim=1000`.
   - Resultado esperado: HTTP 400; mensagem clara de validação; nenhum registro inserido.

3. **Bloqueio de duplicidade diária**
   - Contexto de entrada: já existe registro ativo para mesmo representante e mesma data.
   - Ação: enviar novo `POST /api/km-logs` para a mesma data.
   - Resultado esperado: HTTP 409 ou 400; mensagem informa registro diário já existente; registro original permanece inalterado.

4. **Listagem filtrada por representante**
   - Contexto de entrada: representative A e representative B possuem registros no mesmo período.
   - Ação: representative A executa `GET /api/km-logs?from=YYYY-MM-DD&to=YYYY-MM-DD`.
   - Resultado esperado: resposta contém apenas registros do representative A.

5. **Bloqueio de edição cruzada**
   - Contexto de entrada: representative A tenta alterar registro pertencente ao representative B.
   - Ação: enviar `PUT /api/km-logs/[id_do_B]`.
   - Resultado esperado: HTTP 403 ou 404; registro do B permanece inalterado.

6. **Admin/manager filtrando por representante**
   - Contexto de entrada: usuário admin ou manager autenticado no workspace.
   - Ação: executar `GET /api/km-logs?user_id=<representante>&from=YYYY-MM-DD&to=YYYY-MM-DD`.
   - Resultado esperado: resposta contém apenas registros do representante informado dentro do workspace.

7. **Soft delete**
   - Contexto de entrada: registro ativo existe.
   - Ação: enviar `DELETE /api/km-logs/[id]`.
   - Resultado esperado: HTTP 200; campo `deleted_at` preenchido; registro não aparece em novo `GET /api/km-logs`.

8. **UI mobile 375px**
   - Contexto de entrada: tela `/dashboard/km` aberta em largura 375px.
   - Ação: abrir formulário, preencher campos, salvar e visualizar listagem.
   - Resultado esperado: formulário em coluna única; cards mobile; botões com área mínima de 44px; sem overflow horizontal.

9. **UI tablet 768px**
   - Contexto de entrada: tela `/dashboard/km` aberta em largura 768px.
   - Ação: usar filtros de período e salvar um registro.
   - Resultado esperado: layout intermediário coerente; filtros acessíveis; cards/tabela conforme breakpoint definido.

10. **UI desktop ≥1024px**
    - Contexto de entrada: tela `/dashboard/km` aberta em desktop.
    - Ação: listar registros com filtros e editar um item.
    - Resultado esperado: tabela desktop preservada; sidebar funcional; nenhum layout existente quebrado.
