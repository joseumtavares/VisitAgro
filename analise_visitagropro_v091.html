1 · Resumo Executivo

O log de build da Vercel foi interrompido imediatamente após "Linting and checking validity of types ..." — o que indica que o processo falhou durante a etapa de type-checking do TypeScript, e não durante compilação ou bundling. O build passou em Compiled successfully, mas travou no lint/type phase.

A causa raiz imediata identificada no código é o arquivo src/app/dashboard/map/page.tsx que usa useState sem importá-lo. Isso produz um erro TypeScript fatal (ReferenceError em nível estático) que derruba o build. Há um segundo problema crítico no mesmo arquivo: useEffect retornando null explícito, que viola o tipo de retorno esperado (void | (() => void)).

Além do bloqueante de build, foram encontrados: um hash_algo inválido no Schema_Fix_user.sql (valor 'pbkdf2' não está no CHECK constraint do banco, quebraria o login do admin); RLS desabilitado globalmente no schema; e o package.json com versão desatualizada (0.8.1). As demais correções da v0.9.1 (JWT HMAC, apiFetch, race condition de order_number) estão corretamente implementadas.

Evidência O arquivo importa apenas useEffect da linha 3, mas a linha 19 usa useState (não importado). O log de build confirma: build passou em "Compiled successfully" mas travou em "Linting and checking validity of types..."

B-2 useEffect retornando null — tipo de retorno inválido (TypeScript strict) BLOQUEANTE
Localização src/app/dashboard/map/page.tsx : linha 24
src/app/dashboard/products/page.tsx : linha 36
Evidência Com "strict": true no tsconfig, o tipo de retorno de um callback de useEffect é void | (() => void). Retornar null é um erro de tipo: Type 'null' is not assignable to type 'void | (() => void)'
Causa raiz Pattern de guard escrito como if (!hydrated) return null; dentro do useEffect, mas null não é um retorno válido — só void ou cleanup function 
// ATUAL — erro TypeScript
useEffect(() => {
  if (!hydrated) return null;  // ← TYPE ERROR
  if (!isAuthenticated) { router.push('/auth/login'); return; }
}, [isAuthenticated, router]);

FIX-1 Corrigir map/page.tsx — adicionar useState e remover return null nos useEffects
Arquivo
src/app/dashboard/map/page.tsx
Trecho Atual
import { useEffect } from 'react';
...
const [hydrated, setHydrated] = useState(false);
useEffect(() => { setHydrated(true); }, []);
useEffect(() => {
  if (!hydrated) return null;
  if (!isAuthenticated) { router.push('/auth/login'); return; }
}, [isAuthenticated, router]);
Trecho Corrigido
import { useEffect, useState } from 'react';
...
const [hydrated, setHydrated] = useState(false);
useEffect(() => { setHydrated(true); }, []);
useEffect(() => {
  if (!hydrated) return;          // void, não null
  if (!isAuthenticated) { router.push('/auth/login'); }
}, [hydrated, isAuthenticated, router]);
Explicação
Dois ajustes atômicos: (1) adicionar useState ao import, eliminando o ReferenceError fatal. (2) trocar return null por return; (void), compatível com o tipo do callback do useEffect. Também adicionamos hydrated às dependências do segundo effect para corretude do ESLint.
Risco
Mínimo — ajuste puramente sintático, comportamento idêntico
Validação local

C-1 Schema_Fix_user.sql usa hash_algo='pbkdf2' — violaria CHECK constraint do banco CRÍTICO
Localização Schema_Fix_user.sql · schema_completo_v09.sql (CHECK constraint)
Evidência O schema define: CHECK (hash_algo = ANY (ARRAY['sha256','bcrypt'])). O arquivo Schema_Fix_user.sql insere hash_algo = 'pbkdf2' — valor fora do constraint. O Supabase rejeitaria o INSERT/UPDATE com erro de constraint violation. Adicionalmente, auth.ts não tem branch para 'pbkdf2' — nunca conseguiria fazer login com esse hash.
Impacto Usuário admin ficaria sem acesso. O hash pbkdf2:sha512:... no Schema_Fix_user.sql usa formato diferente do que auth.ts espera (formato sha256: com separadores).

C-2 RLS desabilitado globalmente em todas as tabelas — risco de segurança CRÍTICO
Localização schema_completo_v09.sql : linha 454
Evidência O bloco final do schema executa ALTER TABLE % DISABLE ROW LEVEL SECURITY para todas as tabelas (users, clients, orders, commissions, etc). Com a anon key do Supabase exposta no browser (NEXT_PUBLIC_SUPABASE_ANON_KEY), qualquer usuário pode fazer queries diretas via PostgREST sem autenticação.
Contexto O projeto usa auth própria (JWT) e o cliente browser usa anon key apenas (supabase.ts). Como todas as queries reais passam pelo service_role server-side, o RLS desabilitado não afeta o fluxo normal — mas expõe o banco a consultas diretas pela anon key via API REST do Supabase.
Impacto Exposição de dados de clientes, pedidos e comissões via API pública do Supabase (https://seu-projeto.supabase.co/rest/v1/clients)
C-3 Singleton de supabaseAdmin via variável de módulo — comportamento indefinido em serverless BUG POTENCIAL
Localização src/lib/supabaseAdmin.ts : linha 5
Evidência let _client: SupabaseClient | null = null a nível de módulo. Em funções serverless da Vercel, o módulo pode ser recarregado entre invocações ou compartilhado entre invocações quentes. O Supabase SDK com auth desabilitado é stateless para queries, então o risco real é baixo — mas o singleton pode reter estado de auth ou headers em edge cases.
Impacto Risco baixo mas real em picos de tráfego ou restarts do worker. Não afeta queries normais.
Recomendação Manter o padrão atual (é a prática recomendada pelo Supabase para Next.js). Apenas garantir que auth: { autoRefreshToken: false, persistSession: false } esteja sempre presente — já está.

S-2 cleanup/route.ts — delete sem filtro em produção pode perder dados BUG
Localização src/app/api/admin/cleanup/route.ts : linha ~40
Evidência .delete().neq('id', '00000000-0000-0000-0000-000000000000') é um truque para deleção total (UUID fake). Se o Supabase mudar o comportamento de .delete() sem filtro, ou se a política de RLS bloquear, o filtro pode falhar silenciosamente. O comentário inline confirma: "deleta tudo".
Impacto Perda irreversível de todos os dados da tabela em produção se acionado por engano. O PIN de segurança mitiga, mas não elimina o risco.
Recomendação Adicionar variável de ambiente ALLOW_CLEANUP=true e bloquear a rota em produção por padrão. No mínimo, logar o workspace antes de deletar.
